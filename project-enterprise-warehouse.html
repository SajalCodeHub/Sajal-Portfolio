Enterprise Warehouse

A hybrid warehouse architecture handling 1M+ insurance records daily — bulk historical loads via Spark batch, time-sensitive operational data via Kafka streaming, converging in Amazon Redshift with WLM queue isolation, SLA monitoring, and idempotent writes throughout.

Problem

A Single Batch Path That Couldn't Scale

The nightly batch job couldn't satisfy both volume growth and freshness requirements simultaneously.

The existing pipeline was batch-only. As daily insurance record volume grew to 1M+, the nightly job started overrunning its completion window — delaying downstream reporting that had hard start times. At the same time, operational use cases needed data in minutes, not hours. Risk modeling teams were working with data that was up to 24 hours stale.

The solution wasn't "make batch faster" in isolation. It was workload separation: keep Spark batch for high-volume historical loads and complex multi-source joins, introduce Kafka streaming for latency-sensitive operational events, and isolate both paths with Redshift WLM so neither could degrade the other or impact BI analysts running concurrent queries.

Architecture

Hybrid Batch + Streaming Architecture

Two independent ingestion paths converge at Redshift with explicit isolation and different write semantics.

Batch Path

Spark Batch — Volume, Joins, Completeness

The batch path handles everything where completeness matters more than freshness.

Source systems write raw data to S3 staging, partitioned by date. Spark reads from S3, performs multi-source joins, deduplication, and schema normalization, then loads into Redshift via COPY for maximum throughput. Airflow orchestrates the schedule with SLA monitoring and retry logic.

  • COPY from S3: Bulk load via COPY is 10–100x faster than row-by-row INSERT for Redshift. Batch files are sized to minimize COPY overhead while respecting Redshift's MPP distribution
  • Spark deduplication pre-load: Duplicates removed in Spark before hitting Redshift — prevents downstream consumers from seeing duplicate records and avoids expensive post-load dedup operations
  • Partition alignment: S3 partitions aligned with Redshift SORTKEY — reduces data movement during scan-heavy queries and preserves MPP parallelism
  • p99 window guarantee: Downstream reporting has hard start times. Airflow SLA monitoring fires on "running late" — not just "failed" — giving time to resolve before the window closes

Streaming Path

Kafka Streaming — Freshness Under 5 Minutes

Operational events where sub-5-minute delivery has real business value — risk modeling and real-time alerting.

Operational event sources publish to Kafka topics. Consumers read micro-batches and write to Redshift staging tables. A controlled merge process then promotes staged records into target tables on a schedule — providing freshness without the performance penalty of row-by-row inserts or COPY lock conflicts with the batch path.

Idempotent Merges
Kafka delivers at-least-once. Streaming merges are keyed on stable record identifiers — reprocessing a batch never creates duplicate rows in financial/actuarial tables.
Dead-Letter Topics
Records failing schema validation route to DLQ topics with structured failure reasons. DLQ volume is monitored — spikes indicate source schema drift before it becomes a data loss event.
Consumer Lag Tracking
Consumer lag thresholds alert before a consumer falls critically behind. Lag spikes typically indicate downstream write slowdown or schema mismatch — caught proactively.
WLM Queue Isolation
Streaming merges run in a dedicated WLM queue — separate from batch COPY loads and BI analyst queries. Neither path can saturate the others' concurrency slots.

Reliability

Reliability & Observability

Designed to surface problems before stakeholders notice stale or missing data.

SLA monitoring via Airflow callbacks
SLA miss callbacks fire on "running late" — not just "failed." Slow-running batch jobs get caught before the downstream reporting window closes, giving time to investigate and resolve.
Row count reconciliation
Batch loads reconcile source record counts against Redshift counts per partition. Discrepancies beyond tolerance halt downstream consumption — no silent partial loads reaching BI reports.
Explicit dependency graph
Every transformation declares its upstream dependencies. No step runs on data that hasn't passed reconciliation. Failed reconciliation propagates correctly through the Airflow dependency chain.
AWS CloudWatch alerting
Custom Python validation scripts and CloudWatch alerts monitor for schema drift, late-arriving data, and volume anomalies — catching issues before they impact downstream consumers.

Performance Engineering

+30% Throughput — Without Infrastructure Upgrades

Every gain came from tuning and schema design — not from throwing more hardware at the problem.

Spark partition sizing
Repartitioning aligned task sizes with executor memory and target S3 file sizes — reducing Spark scheduler overhead and improving throughput on the largest daily loads significantly.
Broadcast join optimization
Broadcast hints for small dimension tables eliminated expensive shuffle joins identified in Spark UI execution plan analysis — the single largest driver of the 30% throughput improvement.
Redshift DIST/SORT alignment
Join-heavy tables aligned on DISTKEY; scan-heavy tables received compound SORTKEY patterns — reducing node-to-node data movement and improving BI query speed by 40%.
Vacuum/analyze scheduling
High-churn tables vacuumed during off-peak windows using targeted operations — preserving query performance on tables with high delete/update rates without runaway compute costs.

Results

Measurable Outcomes

  • +30% throughput on the 200GB+ daily Spark batch — achieved through partition tuning and broadcast join optimization, without touching infrastructure or increasing compute spend
  • 40% improvement in BI query performance through star and snowflake schema redesign — DIST/SORT key alignment in Redshift alone cut join-heavy query times significantly
  • Zero duplicates in critical financial and actuarial tables — idempotent merge semantics on stable record identifiers held across both batch and streaming paths
  • AWS IAM access policies, column-level encryption, and data masking in place — required for handling sensitive policyholder records under enterprise governance standards

Tradeoffs

Tradeoffs & What I'd Improve Next

Two paths = two operational surfaces
Batch + streaming doubles failure modes and on-call complexity. A unified Spark Structured Streaming platform with configurable micro-batch intervals could reduce operational surface while preserving the freshness/completeness tradeoff.
Schema evolution is coordinated manually
Source changes require synchronized updates across Spark jobs, S3 schemas, and Redshift table definitions. A schema registry and dbt-managed migrations would make schema evolution safer, auditable, and reviewable via PR.
Row-level lineage requires manual correlation
Lineage exists at the pipeline level. Tracing a specific record from source through Spark to Redshift requires manually correlating logs. A lightweight audit table written at ingestion time would make individual record traceability straightforward without heavy tooling.

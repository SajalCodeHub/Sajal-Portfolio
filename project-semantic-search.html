Problem

Why Keyword Search Was Failing

Search failures were a retrieval infrastructure problem — not a data availability problem.

Product data lived in a relational store with free-text fields. Users searched with exact-match and LIKE queries, which failed completely on synonyms, abbreviations, typos, and natural language phrasing. The same product would return no results under one name and dozens under another. Staff spent 3–5 hours per week resolving search misses manually — looking up products, correcting queries, and manually matching results.

Vector search was the right approach because it increases recall under imperfect and varied input. The goal was operational: return the right product fast enough to be used interactively, with the reliability behaviors — fallbacks, monitoring, evaluation gates — that production systems require. This could not be a prototype that worked in a notebook but broke under real usage patterns.

Architecture

Two-Path Design: Offline + Online

Separating indexing from querying was the most important architectural decision — it's what makes the system resilient.

The offline and online paths are fully decoupled. The indexing DAG can fail, be rerun, or be paused for maintenance without affecting query availability. This was a deliberate design choice — a reindexing failure should not mean zero search results for users.

Offline Path

The Indexing Pipeline

Scheduled, incremental, and designed to handle catalog changes without full rebuilds.

The indexing pipeline runs on an Airflow schedule — reading product records from the Gold layer, preparing text for embedding, calling the Gemini embedding API, and upserting vectors into PostgreSQL pgvector on product_id. Upserts allow incremental indexing without full rebuilds when only a subset of products changes.

  • Incremental upserts: Only changed products are re-embedded. The catalog changes daily — a full rebuild each run would be wasteful and introduce unnecessary API costs
  • Text preparation step: Product text is normalized, synonyms expanded, and long descriptions truncated before embedding — text quality directly determines retrieval quality
  • Embedding consistency: All vectors use the same model version (text-embedding-004) and text prep pipeline. Model version changes require a full rebuild triggered intentionally
  • Staleness monitoring: The pipeline records a last-indexed timestamp per product. Staleness beyond threshold triggers an alert before users notice degraded results
  • HNSW index: Approximate nearest neighbor via HNSW provides sub-millisecond retrieval at 1,700 records — significantly faster than exact IVFFlat at this scale

Online Path

The Query Path

Fast, observable, and designed to fail gracefully — not silently.

Query Embedding
User query text goes through the same preparation pipeline as product text before embedding. Embedding is the dominant latency contributor at ~60% of p95 end-to-end time. A timeout on the Gemini API call triggers the trigram fallback — users always get results, even when the embedding endpoint is degraded.
pgvector Similarity Search
Cosine similarity search against the HNSW index returns Top-K candidates with their similarity scores. At 1,700 vectors, retrieval is sub-millisecond. The similarity score is surfaced alongside results to give the caller context on retrieval confidence.
Trigram Fallback
When embedding fails (API timeout, rate limit), the system automatically falls back to PostgreSQL trigram search. The response includes a search_mode field indicating which path was used — making degraded behavior explicit and observable rather than silent.

Observability

Production Monitoring

Designed to surface problems before users find them.

Index Freshness
Last-indexed timestamp monitored per product. Staleness beyond configured threshold triggers an alert — degraded results surface proactively, not after a user reports a search miss.
Score Distribution
Top-1 similarity scores tracked across queries. A sustained drop in average scores indicates embedding drift, text prep regression, or catalog changes that need attention.
Latency SLOs
p50 / p95 / p99 query latency logged. A p95 spike typically indicates Gemini endpoint degradation before error rates rise — giving time to respond before user experience degrades.
Fallback Rate
Trigram fallback activation rate tracked separately. A spike in fallback usage signals embedding API instability before it becomes an outage — allowing proactive response.

Evaluation

Deployment Gates Based on Retrieval Quality

Not "it feels better" — a fixed evaluation set that runs before any indexing change or model update ships.

A fixed set of 120 query–product pairs was assembled from historical search logs and staff-verified matches. This evaluation set runs before every indexing change, text prep update, or embedding model version change is deployed to production. Regressions block deployment.

Recall@k
Primary metric: does the correct product appear in top-k results? Recall@3 and Recall@5 are both tracked. A Recall@3 regression below the configured threshold blocks deployment. Currently maintained above 90% through catalog changes and text prep iterations.
MRR (Mean Reciprocal Rank)
Measures average rank position of the correct result. Useful when recall is stable but ranking quality degrades — catching cases where the right product is always in the results but ranked 4th instead of 1st, which still degrades user experience.
Failure Case Analysis
Every query missing the correct result in top-5 is reviewed manually. Most failures traced to text preparation — missing synonyms, overly aggressive truncation — rather than embedding model quality. This guided iterative improvements to the text prep pipeline.
Latency Regression Fixture
The same 120-pair evaluation set runs as a p95 latency baseline. Consistent load against a fixed dataset allows apples-to-apples performance comparison across model versions, infrastructure changes, and index size growth.

Results

Measurable Outcomes

  • 3–5 hours per week of manual product lookup work eliminated — operations staff no longer spend time resolving search misses by hand
  • Zero silent failures — every degraded query is explicitly flagged via search_mode field, so fallback behavior is always observable rather than invisible
  • Index staleness alerts surfaced and resolved multiple times before any user-visible quality degradation — the monitoring caught real problems before users did
  • KWI's first AI-powered system in production — established that RAG is viable on the existing data infrastructure, which opened the door for the feedback classification work that followed

Tradeoffs

Tradeoffs & What I'd Improve Next

Embedding latency is the bottleneck
Query-time embedding is ~60% of end-to-end latency. Next step: cache frequently embedded queries, or move to a locally hosted embedding model to eliminate the external API dependency and reduce p95 significantly.
Single-stage retrieval
Current ranking is pure vector similarity. A lightweight re-ranker — cross-encoder or business rules on structured product fields — would improve precision when multiple candidates score similarly close.
Evaluation set coverage
The 120-pair set covers common patterns well but misses long-tail edge cases. Logging implicit feedback (which result the user actually selected) would expand ground truth organically without manual labeling effort.
pgvector at scale
HNSW on PostgreSQL performs excellently at 1,700 records. At 100K+, memory and latency would need re-assessment. The indexing pipeline is designed so swapping the vector store (to Pinecone, Weaviate, etc.) doesn't require rewriting any pipeline logic.

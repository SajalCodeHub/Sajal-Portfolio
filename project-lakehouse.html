Problem

What Needed Solving

Data was arriving from multiple operational systems — MySQL, REST APIs, SFTP — with inconsistent schemas, no enforced contracts, and no observability into failures. Downstream BI models required frequent manual correction due to schema drift, silent duplicates, and partial loads that completed without error but delivered incomplete data.

There was no Airflow orchestration layer, no data quality framework, and no dimensional model. The objective was to build the entire data infrastructure from scratch — layered contracts, isolated failure domains, deterministic backfills, and a modeling layer that downstream teams could rely on without understanding the pipeline internals.

On top of that, leadership wanted to explore AI-augmented capabilities. The platform needed to be designed from day one to support both BI consumers and AI retrieval systems as first-class outputs.

Architecture

Medallion Architecture Design

Three layers, each with a single responsibility and a hard contract boundary at every promotion step.

Bronze Raw Landing — Append-only, Auditable
Raw data lands exactly as received — no transformation, no filtering. Schema validation runs at arrival to catch structural breaking changes early. Each partition is append-only and immutable. If Silver logic changes, Bronze can always be reprocessed from scratch without data loss.
Silver Cleansing + Enrichment — Contract-enforced
Deduplication, null filtering on critical fields, type casting, and business rule application. Data quality gates run here — schema integrity, null-rate thresholds, record completeness, and referential consistency checks. Failures block promotion to Gold rather than silently passing bad data downstream.
Gold Dimensional Models — BI and AI Ready
Star and snowflake schema design in PostgreSQL and Snowflake, built with dbt for version-controlled, tested transformations. Gold tables are the single source of truth for 5+ Power BI dashboards and the embedding pipeline that feeds the AI semantic search system.

Orchestration

Airflow Architecture — Built from Scratch

15+ DAGs designed with production-grade reliability from day one — not retrofitted after the first incident.

The entire Airflow orchestration layer was designed and built from scratch. This meant defining the DAG architecture, dependency chains, retry policies, SLA alerting, and failure recovery patterns before a single pipeline went live. The result: 99%+ uptime with zero unplanned data delivery failures since launch.

  • SLA callbacks per DAG: Every pipeline has a defined SLA. Violations alert immediately — the engineering team finds out before downstream consumers notice missing data
  • Retry policies with backoff: Transient failures (network timeouts, API rate limits) are automatically retried with configurable backoff. Only true failures surface as alerts
  • Dead-letter handling: Malformed records are routed to a dead-letter partition for inspection rather than silently dropped or causing pipeline failure
  • Safe backfill patterns: All DAGs are idempotent and partition-scoped. Historical reprocessing can be triggered without risk of data duplication
  • Explicit dependency graphs: Every upstream/downstream relationship is modeled in the DAG. No implicit dependencies that break silently when execution order changes

dbt Modeling

Analytical Model Foundation

Modular, version-controlled SQL transformations that teams can trust and audit.

dbt was chosen for its version-controlled SQL, built-in testing framework, and the ability to generate documentation automatically. Models are organized into staging and mart layers — staging handles type casting and light enrichment, marts implement the dimensional design that powers dashboards and analytical consumers.

Staging Models

Type casting, column renaming, and light enrichment from RAW sources. One model per source table — clean separation of concerns.

Mart Models

Star schema facts and dimensions — business logic, aggregations, and joins implemented once and reused across all consumers.

dbt Tests

Unique, not_null, accepted_values, and relationship tests run on every model. Test failures block deployment.

dbt Docs

Auto-generated lineage graphs and model documentation. Every column described. New team members can understand the model in minutes.

Data Quality

Quality Enforcement — Not Quality Theater

Checks that block bad data from moving forward — not checks that generate reports nobody reads.

Data quality checks run as first-class Airflow tasks integrated into the pipeline flow. A failing check blocks promotion to the next layer. The result is a 95%+ anomaly catch rate before any bad data reaches Power BI or the AI system.

Schema Integrity
Column presence, data type validation, and format checks run at Bronze ingestion. Schema-breaking changes are caught before they corrupt downstream layers.
Null Handling
Null-rate assertions on critical fields with configurable thresholds. High null rates in key columns surface as alerts rather than silent data quality degradation.
Record Completeness
Row count expectations per partition — unexpected drops surface late-arriving data, source system issues, or pipeline failures before BI consumers notice missing records.
Referential Consistency
Relationship integrity checks between fact and dimension tables. Orphaned records are caught at the Silver gate rather than surfacing as broken dashboard filters.
Duplicate Detection
Primary key uniqueness enforced before merge operations. Safe deduplication logic preserves the most recent record while logging discarded duplicates for audit.
Pre-Release Gates
All checks run automatically before each pipeline deployment. New logic cannot reach production without passing the full validation suite — catching 95%+ of anomalies before impact.

AI Integration

Extending the Platform Into AI

The lakehouse wasn't just a BI platform — it became the data foundation for KWI's first AI-powered system.

Once the Gold layer was stable and trusted, it became the foundation for two AI-augmented capabilities. Both were designed as extensions of the existing pipeline infrastructure — not standalone experiments.

Semantic Search Pipeline
An Airflow DAG reads cleaned product records from the Gold layer, generates embeddings via Google Gemini Vertex AI, and upserts them into PostgreSQL pgvector. This became KWI's first AI-powered product search system — delivering sub-second similarity search across 1,700+ products. The Gold layer's data quality directly determines embedding quality.
Product Feedback Classification
Azure OpenAI integrated into a live production pipeline to automate product feedback classification and enrichment. Previously manual — taking 3–5 hours per week — now runs automatically on each pipeline cycle, enabling real-time data-driven product decisions.

Results

Measurable Outcomes

  • Zero unplanned data delivery failures since launch — the SLA alerting and retry architecture means incidents are caught and resolved before downstream teams notice anything wrong
  • ~50% improvement in dashboard query response times through SQL workload redesign and a targeted indexing audit — no infrastructure changes, just better queries
  • First AI-powered system at KWI shipped — the semantic search pipeline and feedback classification both run on the Gold layer data, which means data quality directly determines AI quality
  • 3–5 hours/week of manual product analysis work eliminated — Azure OpenAI feedback classification now runs automatically on every pipeline cycle

Tradeoffs

Tradeoffs & What I'd Improve Next

Centralized contract registry
Schema contracts are currently enforced per pipeline. A shared registry would improve discoverability and reduce duplication when onboarding new sources — especially as the number of pipelines grows beyond 15.
Column-level lineage
Row-level lineage exists through dbt. Tracing individual fields across transformations still requires manual inspection of model code. Automated column-level lineage tooling would significantly reduce debugging time for data issues.
Performance regression automation
Performance baselines are documented. A scheduled benchmark suite running against production query patterns would catch regressions before they become visible to dashboard consumers — currently this relies on monitoring and alerting after the fact.

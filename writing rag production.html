Operationalizing RAG: From Prototype to Production Pipeline

The demo worked perfectly. Then I thought about what it looks like at 3am when the embedding API is down and the index is a week stale. That question is where the real design work starts.

What Changes When You Add the Word "Production"

A RAG prototype in a notebook has one requirement: produce relevant results for the queries you tested. A production RAG pipeline has about fifteen more, none of which were on your mind when the prototype was working.

The gap isn't mostly technical. It's mostly operational. A notebook doesn't need a plan for when the embedding model changes. It doesn't need to handle the case where a product is added to the catalog between index refreshes. It doesn't need to tell you when search quality is degrading. It doesn't need to keep working when the external API returns 429s.

When I shipped KWI's semantic search system, the prototype took about a week. The production system — with monitoring, index freshness checks, fallback modes, evaluation gates, and an embedding refresh pipeline that runs on schedule — took considerably longer. That difference is what this piece is about.

The Five Things That Break First

These are the specific failure modes I designed for — and in some cases discovered — while building the production system. In rough order of how soon after launch each one became relevant.

Index staleness. Your catalog changes daily. New products are added. Descriptions are updated. Discontinued items should stop appearing in results. If your index refresh pipeline fails silently, users start seeing results that don't reflect reality. Freshness monitoring isn't optional. You need to know the age of your index at all times, and you need an alert when it exceeds a threshold that users would notice.

Embedding API outages. External embedding APIs go down. They throttle under load. They change pricing models and return errors you didn't test for. The question isn't whether this will happen but what your system does when it does. Serving stale results is better than serving nothing. A trigram fallback is slower and less accurate but better than a 500 error. Your fallback strategy should be explicit, tested, and surfaced in your response metadata so you always know when it's active.

Latency degradation that isn't a failure. A p95 query time that drifts from 400ms to 750ms to 1.1 seconds over three weeks won't trigger any error alerts. The system is working. It's just getting slower in a way that will eventually break the interactive UX threshold without any single event causing it. The only way to catch this is a latency baseline fixture — a fixed set of queries you run consistently so you can see the trend.

Query distribution shift. The queries users actually send are different from the ones you tested. Your evaluation set covers what you predicted. Production users find the edge cases you didn't. Building a mechanism to log queries (stripped of PII) and periodically review failure cases is how you find out where your system actually breaks — not where you assumed it might.

Text preparation assumptions that don't scale. The text cleaning logic that worked on your test set starts failing on edge cases in the full catalog: product names with special characters, descriptions in multiple languages, fields that are sometimes empty. These aren't embedding model problems. They're text pipeline problems. And they're much easier to fix when you've been logging the specific records where retrieval is failing.

Designing the Offline Indexing Path

The offline indexing pipeline is where most of the reliability engineering lives. It runs on a schedule, reads product records from the data warehouse, generates embeddings, and upserts them into pgvector. Getting this path right determines whether the system stays fresh or silently degrades.

The key design decision: upserts, not full rebuilds. Rebuilding the entire index on every run means any failure during the run leaves you with a partially updated index. Upserting by record ID means each run is idempotent — if it fails halfway through, you run it again and it picks up where it left off without duplicating any records.

The second decision: track index age explicitly. After every successful run, write a timestamp to a metadata table. Your monitoring queries that table, not the embedding store. If the timestamp is more than N hours old, fire an alert. This catches failed runs, stuck pipelines, and misconfigured schedules before any user notices degraded results.

The third decision: separate the embedding generation from the index write. If embedding generation fails (API timeout, rate limit), the previous good embeddings are still in the index. You don't want a partial API failure to take down the search system. Batch the embedding calls, handle failures per-record, and write to the index only after a full batch succeeds.

Designing the Fallback Mode

Every production search system needs a fallback for when the primary path is unavailable. For vector search, the natural fallback is traditional text search — in our case, trigram similarity via PostgreSQL's pg_trgm extension.

The important part isn't just having a fallback — it's making the fallback transparent. Every search response in our system includes a search_mode field that tells the caller whether the result came from vector similarity or trigram fallback. This means: monitoring can track how often fallback is active, engineers can see at a glance whether a degraded response was a fallback result, and users aren't silently served worse results with no indication that anything changed.

Hidden fallback behavior is how you end up in a situation where the embedding API has been down for two days and nobody noticed because the trigram results looked plausible. Explicit mode surfacing in the response prevents that.

Building an Evaluation Set That Stays Useful

An evaluation set is only useful if it reflects the queries your system actually handles. A 120-pair test set built at launch will cover the patterns you predicted — but won't cover the edge cases that real usage surfaces over time.

The way to keep it relevant: log queries where the top result wasn't selected (or where the user rephrased and searched again). These implicit failure signals, stripped of PII, are ground truth for where your system is actually falling short. Periodic review of these cases — even just 20 minutes a week — will surface patterns faster than any synthetic test set.

For the latency fixture specifically: keep a fixed set of queries and run them on a schedule against the live system. The value isn't correctness — it's consistency. The same queries run against the same index size over time give you an apples-to-apples comparison when you change the embedding model, upgrade pgvector, or increase the catalog size. Without that baseline, you're guessing at whether a performance change was caused by your last deploy or by something else that happened the same week.
